# Улучшения существующих систем

## pr-app/back

### Данные и схема

- **Нет ни одного денежного поля в `PurchaseRequest`.** Без суммы
  невозможна сверка requested/invoiced/paid (боль №4 из README). Добавить
  `estimated_amount: Numeric(12, 2)`, nullable — миграция Alembic, сверка по сумме работает только для PR, созданных
  после миграции; старые PR остаются без суммы и не участвуют в
  reconciliation.
- **Схема создаётся через `Base.metadata.create_all` в `scripts/init_db.py`,
  Alembic не используется**, хотя `alembic` в зависимостях не настроен.
  Любое изменение схемы сегодня требует пересоздания БД с нуля. Ввести
  Alembic с первой ревизией - текущая схема, дальше — миграции, в том
  числе для `estimated_amount` и будущего `supplier_id`.
- **Поставщик хранится как free-text** на каждой записи — источник боли
  №3. После появления `procurement-platform` — заменить на `supplier_id`
  (просто `int`, без физического FK на чужой сервис — валидация через
  вызов `GET /suppliers/{id}`).

### Бизнес-логика

- **`_next_code()` генерирует код PR через `MAX(id) + 1`** — комментарий в
  коде сам признаёт "not concurrency-safe". Два параллельных запроса на
  создание PR могут получить одинаковый код. Исправление: либо положиться
  на autoincrement `id` БД и форматировать код из него постфактум
  (`f"PR-{id}"` после `INSERT ... RETURNING id`), либо использовать
  Postgres `SEQUENCE` явно.
- **Статусная машина `_can_transition` — набор `if/else` прямо в роутере**
  (`routers/purchase_requests.py`). Нет единого места, где видны все
  допустимые переходы, нет unit-тестируемости в изоляции от HTTP-слоя.
  Вынести в explicit transition table (`dict[Status, set[Status]]`) или
  `enum` + dataclass-стейт-машину, покрыть тестами отдельно от роутера.

### Безопасность

- **`pr_sessions`: токен `secrets.token_urlsafe`, без TTL и без эндпоинта
  ревокации.** Сессия живёт вечно, выйти "по-настоящему" нельзя. Добавить
  `expires_at` + фоновую очистку просроченных строк, плюс `POST /logout`,
  инвалидирующий конкретный токен.
- **Только два значения роли (`employee`, `finance`), без таблицы ролей.**
  Любая дальнейшая гранулярность (например, отдельная роль "закупщик" для
  работы с ERP) сегодня требует менять enum в коде. Ввести отдельную
  таблицу `roles`/`permissions`, даже если на старте в ней будут те же два
  значения — точка расширения нужна уже сейчас, поскольку ERP-роль скоро
  появится по факту.

### Тесты и CI

- **В репозитории `pr-app/back` не найдено ни одного теста.** `pyproject.toml`
  настраивает `ruff`, есть `pyrightconfig.json` — то есть инструменты для
  линтинга/тайпчекинга заведены, но не используются автоматически. Добавить
  `pytest` с покрытием хотя бы для `_can_transition` и генерации кода (тест
  на конкурентность — два параллельных `INSERT`, проверка уникальности
  кодов). Добавить GitHub Actions workflow: `ruff check`, `pyright`, `pytest`
  на каждый PR.
- **Синхронный SQLAlchemy в асинхронном FastAPI-приложении.** Каждый
  блокирующий вызов к БД блокирует event loop целиком — под нагрузкой это
  убивает throughput сильнее, чем кажется на тестовом объёме. Миграция на
  SQLAlchemy 2.0 async + `asyncpg` — состыковывается с реальным стеком
  кандидата и является логичным следующим шагом независимо от интеграции.

## invoice-app/back

### Данные и схема

- **`purchaseRequestNumber` — обычная `String`, без FK**, что и
  комментирует сам код ("there's no foreign key back to the PR app's
  purchase_requests table"). В рамках интеграционного дизайна поле
  остаётся free-text (см. `DIAGRAM.md`), но сверка переносится на сторону
  `procurement-platform`/Hub — не пытаемся синхронно валидировать его на
  каждый ввод, это отдельная зона ответственности.
- **PDF хранится как `bytea` напрямую в строке инвойса.** Это раздувает
  таблицу и бэкапы, усложняет full-table scan. Перенести в объектное
  хранилище (S3-совместимое, например MinIO для докер-компоуза) и хранить
  только ссылку/ключ.
- **Поставщик — free-text**, как и в pr-app. Тот же фикс: `supplier_id` из
  `procurement-platform`.
- **Схема Hibernate (`ddl-auto`) вместо версионированных миграций.**
  Аналог проблемы Alembic в pr-app, только для Java-стека — ввести Flyway
  или Liquibase.

### Безопасность

- **`InvoiceController` напрямую читает таблицу `users`** из общей БД
  (`@Table(name = "users")`, помечено как read-only view) для определения
  роли при логине. Это и есть текущий shared-database anti-pattern —
  межсистемная зависимость через таблицу, а не контракт. Не трогается в
  этом раунде интеграции (отдельная хирургия, выходящая за рамки текущего
  PR), но явно фиксируется как технический долг: в будущем — заменить на
  вызов `GET /whoami` к pr-app вместо прямого чтения его таблицы.
- **Только роль `finance`, без админ-интерфейса и audit trail** —
  аналогично pr-app. Добавить append-only таблицу `invoice_audit_log`
  (кто, когда, что изменил) — внутренний аудит конкретного приложения,
  отдельно от межсистемной проекции Hub.

### Тесты и CI

- Аналогично pr-app — нет видимых тестов; добавить JUnit-покрытие бизнес-
  логики (расчёт `invoice_sum_paid`, доступ по ролям), плюс CI-пайплайн
  (Maven/Gradle build + test на каждый PR).
