# Дорожная карта

Порядок определён зависимостями, риском и YAGNI (не строим то, для чего нет
подтверждённой потребности — например, Kafka/CDC, см. `README.md`).

1. **Supplier registry** (Reference) — фундамент для всего остального,
   низкий риск, мгновенная польза: дубли поставщиков исчезают сразу.
2. **Team + Budget** (Reference) — тот же сервис. Резерв бюджета при
   отправке PR закрывает часть боли №2 с максимальной ценностью на
   единицу сложности.
3. **Procurement hub v1, polling** — единый дашборд для finance, закрывает
   боль №1 без ожидания ERP.
4. **Сумма PR + сверка requested/invoiced/paid** — нужны и поле в PR, и
   Hub из п.3, поэтому не раньше.
5. **ERP-адаптер и PO matching** — ERP пока не подключена; строить раньше
   её появления — speculative engineering. `EventSource` уже готов принять
   новый источник без переписывания Hub.
6. **Уведомления вместо polling сотрудником** — технически проще п.5, но
   ниже по бизнес-ценности, поэтому позже.
7. **RBAC / admin UI / audit trail в каждом приложении** — не блокирует
   межсистемные пайны, может вестись параллельно отдельной командой.
8. **Миграция polling → Kafka/CDC** — только при реальном росте объёма
   или появлении третьего потребителя тех же событий, не по умолчанию.
9. **Отчётность и аналитика** — read-side, выигрывает от того, что к этому
   моменту через Hub уже прошла история чистых сверенных данных.
