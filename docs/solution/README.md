# Procurement integration — design proposal

Данный каталог является решением задач из корневого README: дизайн интеграции
PR ↔ Invoice ↔ ERP, список улучшений каждой существующей системы и дорожная
карта новых возможностей. Диаграммы — в `DIAGRAM.md`, эндпоинты и схемы —
в `API.md`, доработки существующего кода — в `IMPROVEMENTS.md`, план фич —
в `ROADMAP.md`.

Ни одна из пяти болей в корневом README не описывает отсутствие канала
сообщений между pr-app и invoice-app. Они описывают сущности, которых не
существует ни в одной системе: Supplier, Team, Budget и PurchaseOrder
сегодня не принадлежат ни pr-app, ни invoice-app. Брокер сообщений или
любой другой механизм доставки между двумя системами решает задачу
передачи данных, которые уже где-то существуют — здесь передавать нечего.
Решение вводит новый сервис, `procurement-platform`, который становится
владельцем этих сущностей и единой точкой, где статус закупки — от заявки
до оплаты — виден целиком.

Сервис состоит из двух модулей с разными требованиями к консистентности.
Reference-модуль (поставщики, команды, бюджеты) — синхронный CRUD: при
отправке PR система должна сразу знать, хватает ли бюджета команде.
Hub-модуль агрегирует статус из обеих систем и в перспективе из ERP; эта
проекция eventually consistent — задержка в несколько секунд в дашборде
finance допустима. Оба модуля развёрнуты в одном процессе, разделены на
уровне Postgres-схем (`reference`, `hub`), а не на уровне отдельных
сервисов: при текущем объёме данных сетевой вызов между ними добавил бы
точку отказа без выгоды. Разделение схем с самого начала делает возможный
будущий перенос в отдельные сервисы механической операцией, а не
переписыванием.

Доставка событий в Hub абстрагирована интерфейсом `EventSource` (см.
`DIAGRAM.md`). Реализация для прототипа, `PollingSource`, опрашивает
`GET /purchase-requests?since=` и `GET /invoices?since=` с фиксированным
интервалом — без брокера сообщений и без новой инфраструктуры. Это
соответствует фактической нагрузке: несколько заявок в день, не поток с
высоким throughput. `KafkaSource` (CDC через Debezium с публикацией в
Kafka) — вторая реализация того же интерфейса, документированная, но не
реализованная: её стоит строить при росте объёма или при появлении
второго потребителя тех же событий, не раньше — иначе операционная
стоимость (Kafka, Zookeeper/KRaft, мониторинг лага консьюмеров) не
покрывается реальной потребностью.

Резервирование бюджета — fail-open: если `procurement-platform`
недоступен в момент отправки PR, заявка всё равно принимается, с флагом
"бюджет не проверен". Блокировка отправки PR недоступностью нового
вспомогательного сервиса без истории SLA создаёт более высокий риск для
основного процесса согласования, чем временная потеря контроля над
бюджетом. Сопоставление PR, Invoice и PO автоматическое при точном
совпадении кода, с флагом на ручную проверку при несовпадении: поле
`purchaseRequestNumber` остаётся текстовым (изменение формы ввода вне
скоупа), поэтому полностью автоматическое сопоставление не гарантируется.

Вне скоупа в этой версии: авторизация между сервисами — общий секрет в
заголовке (`X-Internal-Api-Key`), не OAuth2/mTLS; изменения ролей и
доступа внутри pr-app и invoice-app описаны отдельно в `IMPROVEMENTS.md`;
высокая доступность `procurement-platform` не требуется, пока сервис не
хранит критичные для бизнеса данные, и становится риском, когда в нём
появятся реальные бюджеты и PO.

Запуск — отдельный контейнер в `docker-compose.yml`:

```yaml
  procurement-platform:
    build: ./procurement-platform
    environment:
      - DATABASE_URL=postgresql+asyncpg://postgres:postgres@db:5432/procurement
      - INTERNAL_API_KEY=${INTERNAL_API_KEY}
      - PR_APP_BASE_URL=http://pr-app-back:8000
      - INVOICE_APP_BASE_URL=http://invoice-app-back:8080
    depends_on: [db]
    ports: ["8090:8000"]
```

Структура сервиса — границы модулей без реализации:

```
procurement-platform/
├── reference/
│   ├── models.py        # Supplier, Team, Budget, BudgetReservation — schema="reference"
│   ├── router.py         # /api/v1/suppliers, /teams, /budgets
│   └── service.py        # резервирование бюджета, идемпотентность через unique(budget_id, reference)
├── hub/
│   ├── models.py          # проекция — schema="hub"
│   ├── sources/
│   │   ├── base.py         # абстракция EventSource
│   │   ├── polling.py       # v1: периодический REST-опрос pr-app/invoice-app
│   │   └── kafka.py         # v2 (не реализовано): CDC через Debezium
│   ├── erp_adapter.py      # REST/webhook со стороны ERP
│   ├── matcher.py           # сопоставление PR/Invoice/PO + флаг "needs review"
│   └── router.py             # /api/v1/overview — дашборд для finance
├── db.py
└── main.py
```
