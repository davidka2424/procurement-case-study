# Procurement integration — design proposal

Этот каталог — ответ на три задачи из корневого README: дизайн интеграции
PR ↔ Invoice ↔ ERP, список улучшений каждой существующей системы и дорожная
карта новых возможностей.

## Навигация

## Идея

У существующих болей нет общего владельца: `Supplier`, `Team`, `Budget` и
`PurchaseOrder` — это сущности, которые не принадлежат ни `pr-app`, ни
`invoice-app`, потому что они в принципе межсистемные. Решение в виде связующего звена для передачи сообщений не решит все проблемы, а новый сервис **`procurement-platform`**, который
становится каноническим владельцем этих сущностей и единой точкой сверки
статуса закупки от заявки до оплаты может решить описаннные проблемы.

## Ключевые архитектурные решения

1. **Один сервис, два модуля** Внутри `procurement-platform`
   чётко разделены `reference/` (синхронный CRUD: поставщики, команды,
   бюджеты — каждый запрос требует немедленного консистентного ответа) и
   `hub/` (асинхронная проекция статуса закупки — сверка естественно
   eventually consistent). Разделение на уровне кода и Postgres-схем
   (`reference`, `hub`). Причина объединения в один
   процесс заключается в объёме этой задачи. Если
   команда-владелец, нагрузка или SLA когда-нибудь разойдутся — разнести
   схемы на отдельные базы/сервисы механически просто, именно ради этого
   разделение и сделано на уровне схем заранее.

2. **Доставка событий в Hub — поллинг/вебхук в v1, Kafka/CDC — опциональный
   путь масштабирования** Решение спрятано за
   интерфейсом `EventSource` (см. `DIAGRAM.md`, диаграмма компонентов).
   v1-реализация — `PollingSource`, читающая `GET /purchase-requests?since=`
   и `GET /invoices?since=` каждые N секунд: ноль новой инфраструктуры,
   запускается одной строкой в `docker-compose`. `KafkaSource` (Debezium +
   Kafka) документирована как готовый путь на случай, если объём или
   требование near-real-time реально появятся — но не реализована, потому
   что сейчас это решение для масштаба, которого нет.

3. **Бюджетная проверка — fail-open, а не hard-блокер.** Если
   `procurement-platform` недоступен в момент отправки PR, `pr-app`
   пропускает PR с предупреждением "бюджет не проверен", а не блокирует
   сотрудника. Бизнес-процесс не должен зависеть от аплейма вспомогательного
   сервиса жёстче, чем зависит сейчас от его отсутствия.

4. **Матчинг PR ↔ Invoice ↔ PO — автоматический по точному совпадению кода,
   с явным fallback на ручную проверку.** Поля остаются free-text (мы не
   трогаем форму ввода `purchaseRequestNumber`), поэтому 100% автоматического
   сопоставления обещать нечестно. Hub автоматически связывает то, что
   совпадает, и подсвечивает в дашборде то, что не нашло пары — вместо
   того чтобы тихо терять расхождения, как сейчас в Excel.

## Что сознательно не сделано в этой версии

- **Авторизация между сервисами** — общий секрет в заголовке
  (`X-Internal-Api-Key`), не OAuth2/mTLS. Для прототипа соразмерно; в
  продакшене — service mesh или client-credentials flow.
- **Granular RBAC, admin UI, audit trail внутри каждого приложения** — это
  не межсистемная проблема, она остаётся в `IMPROVEMENTS.md` как доработка
  каждого приложения, а не часть интеграционного дизайна.
- **Полная автоматизация сверки** PO/Invoice — описана выше, осознанно
  half-automated.
- **High availability `procurement-platform`** — однажды появится бюджет/
  команды/PO, и сервис станет критичным; на эту версию это вне скоупа,
  явно отмечено как риск.

## Как это запускается (прототип)

Новый сервис добавляется как ещё один контейнер в `docker-compose.yml`:

```yaml
  procurement-platform:
    build: ./procurement-platform
    environment:
      - DATABASE_URL=postgresql+asyncpg://postgres:postgres@db:5432/procurement
      - INTERNAL_API_KEY=${INTERNAL_API_KEY}
      - PR_APP_BASE_URL=http://pr-app-back:8000
      - INVOICE_APP_BASE_URL=http://invoice-app-back:8080
    depends_on: [db]
    ports: ["8090:8000"]
```

Структура нового сервиса (только архитектура, без реализации — см.
`IMPROVEMENTS.md` про то, почему детальный код здесь намеренно не пишется):

```
procurement-platform/
├── reference/
│   ├── models.py        # Supplier, Team, Budget, BudgetReservation — schema="reference"
│   ├── router.py         # /api/v1/suppliers, /teams, /budgets
│   └── service.py        # резервирование бюджета, идемпотентность через unique(budget_id, reference)
├── hub/
│   ├── models.py          # проекция — schema="hub"
│   ├── sources/
│   │   ├── base.py         # абстракция EventSource
│   │   ├── polling.py       # v1: периодический REST-опрос pr-app/invoice-app
│   │   └── kafka.py         # v2 (не реализовано): CDC через Debezium
│   ├── erp_adapter.py      # REST/webhook со стороны ERP
│   ├── matcher.py           # сопоставление PR/Invoice/PO + флаг "needs review"
│   └── router.py             # /api/v1/overview — дашборд для finance
├── db.py
└── main.py
```
